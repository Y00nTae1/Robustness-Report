# AI 기반 단계별 제작 플레이북 v2 — 비전 보정 폐루프 강건성 검증 과제

이 문서는 텔레픽스 과제(외란 주입 환경 비전 보정 폐루프)를 **AI 에이전트(예: Claude Code)** 로 단계별 제작하기 위한 실행 플레이북이다. 사람이 직접 코딩하는 대신, 각 단계를 4개의 독립 프롬프트(계획 → 코드 → 평가 → 피드백)로 순환시킨다.

---

## ★ v2 변경 이력 (S1 실행 경험 반영)

S1을 실제로 돌리며 드러난 문제를 전 단계에 반영했다. 핵심 3가지:

1. **모든 P2에 "실제 실행 후 증거 첨부" 규칙 추가** — S1 1바퀴가 "미실행"으로 떨어진 원인. 코드만 쓰고 멈추면 실행가능성·산출물 항목이 자동 1점이 된다. 이제 P2는 반드시 이 PC에서 실행하고 콘솔 로그·산출물을 첨부한다.
2. **모든 배경에 "비ASCII 경로 주의" 추가** — 이 PC는 사용자명에 한글(`태원`)이 포함되어, MuJoCo C++ `fopen`과 `cv2.imwrite`가 유니코드 경로에서 깨진다. 우회법(모델: `chdir`+ASCII 파일명, 저장: `cv2.imencode`+파이썬 파일IO)을 전 단계가 따른다.
3. **S1 평가표 "부품이 보임" → "로봇이 보임" 정정** — 베이스 Franka 씬에는 광학 부품이 없다(로봇+바닥뿐). 부품은 S3 이후 등장하므로, S1·S2에서는 로봇 가시성을 기준으로 한다.

또한 S1에서 **확정된 모델 사실**을 부록 B에 정리했다. S2~S6의 P1 "직전 산출물 요약"에 이를 재사용한다.

## 0. 사용 원칙

1. **각 프롬프트는 독립적으로 작동한다.** 프롬프트는 이전 대화 맥락에 의존하지 않는다. 필요한 입력(현재 단계 정의, 직전 산출물, 평가표)을 매번 프롬프트 안에 직접 붙여 넣는다. 새 세션에서 그 프롬프트만 단독 실행해도 동작해야 한다.
2. **단계는 순서대로 진행한다.** S1 → S2 → ... 순서로, 각 단계 안에서 P1 → P2 → P3 → P4 루프를 돈다.
3. **평가표를 통과해야 다음 단계로 간다.** P3 평가가 합격선 미달이면 P4 피드백 후 P2 코드 제작으로 복귀한다. 최대 반복 횟수(권장 3회) 초과 시 한계를 기록하고 다음 단계로 넘긴다.
4. **P2는 반드시 실제 실행하고 증거를 첨부한다.** (v2 신규) 코드 작성에서 멈추지 않는다. 이 PC에서 실행해 콘솔 로그와 생성된 산출물을 결과에 첨부한다. 실행 불가 시 차단 원인을 명시하되, 미실행 상태로 P3에 넘기지 않는다.
5. **모든 산출물은 파일로 저장한다.** 코드는 저장소 구조(아래)에 맞춰 두고, 매 단계 결과를 로그로 남긴다.

### 저장소 구조

```
/external_module   외부 비전 모듈 (M1 Pose 추정, M2 편차·보정)
/sim               MuJoCo 씬·로봇 제어·외란 주입
/experiments       실험 설정·결과 로그·KPI
/report            리포트·Flowchart·결과 영상
/docs              이 플레이북·카드 등 참조 문서 (코드와 분리)
README.md          설치·실행·구조 설명
```

> 주의: 이 플레이북/카드 .md 는 **참조 문서**다. 코드 폴더(/sim 등)와 섞지 말고 /docs 에 둔다. AI 에이전트가 이 문서를 "구현 대상"으로 오해하지 않게 한다.

### ★ 환경 공통 주의 (v2 신규 — 전 단계 적용)

- **이 PC는 비ASCII(한글) 경로 환경이다.** 사용자명에 `태원`이 포함되어 절대경로에 한글이 섞인다.
  - MuJoCo 모델 로드: `mujoco.MjModel.from_xml_path(절대경로)`가 한글에서 깨진다. → 모델 디렉터리로 `os.chdir` 후 **ASCII 파일명만** 넘긴다.
  - 이미지 저장: `cv2.imwrite(한글경로, img)`가 실패한다. → `cv2.imencode(ext, img)` 후 파이썬 `open(path,"wb")`로 바이트를 쓴다.
- **렌더 백엔드:** Windows 기본(wgl)으로 정상 렌더 확인됨. 검은 화면/실패 시 `MUJOCO_GL`(Linux egl/osmesa, Windows glfw) 조정.

## 1. 단계 설정 (Stages)

전체 과제를 7개 단계로 분할한다. 단계는 의존 순서대로 배열되어 있으며, 앞 단계가 통과해야 뒤 단계가 의미를 가진다.

| 단계 | 이름 | 목표 산출물 | 통과 조건(요약) | 상태 |
|------|------|-------------|------------------|------|
| S1 | 환경 부팅 | MuJoCo + Franka 로드, 카메라 RGB(+depth) 렌더, 그리퍼 개폐 | 씬이 뜨고 산출물 저장, 로봇 가시·그리퍼 개폐 확인 | ✅ 통과 |
| S2 | 연동 배관 | 외부 모듈 ↔ 시뮬 JSON 왕복(더미 좌표) | 이미지 전송 → 좌표 수신 1회 성공 | |
| S3 | 비전 모듈 (M1·M2) | Pose 추정 + 편차·보정 벡터 산출 | 부품 위치·각도를 수치로 출력 | |
| S4 | 보정 폐루프 | 집기→이송→보정 반복→허용오차 판정→배치 | 보정 전후 오차 감소가 수치로 확인 | |
| S5 | 외란 주입 | D1 조명·반사 / D2 노이즈 / D3 좌표 드리프트 | 외란 ON 시 이미지·좌표가 실제 변함 | |
| S6 | 실험·로깅 | 외란 강도별 + 보정 ON/OFF 반복 실행, KPI 기록 | KPI 4종이 CSV로 누적됨 | |
| S7 | 리포트 채움 | 측정값·그래프·실패모드로 리포트 자리표시자 채움 | 5개 항목 + 정량 결과 완성 | |

### 단계별 상세 정의

#### S1 — 환경 부팅 ✅
- 목표: MuJoCo에서 Franka Panda 공식 모델을 로드하고, 상단 카메라로 RGB+depth를 렌더하여 파일로 저장한다. 그리퍼 개폐를 확인한다.
- 입력: 없음(시작 단계). MuJoCo Menagerie의 Franka 공식 모델 사용.
- 산출물: `/sim/boot.py`, `/sim/requirements.txt`, RGB/Depth/그리퍼 이미지.
- 제약: 커스텀 URDF 작성 금지. 공식 제공 모델만 사용. 라이브러리는 mujoco, numpy, opencv-python으로 한정.
- 비고: 부품이 아닌 "로봇 가시성"을 기준으로 한다(베이스 씬에 부품 없음).

#### S2 — 연동 배관
- 목표: 외부 모듈(별도 Python 함수/프로세스)이 시뮬 카메라 이미지를 받아 좌표를 되돌려주는 JSON 인터페이스를 뚫는다. 이 단계에서는 비전을 구현하지 않고 더미 좌표를 반환한다.
- 입력: S1 산출물(boot.py의 모델 로드·렌더 함수 재사용).
- 산출물: `/external_module/interface.py`, `/sim/loop.py`(이미지 전송 → 좌표 수신 → 로봇 이동 더미).
- 제약: 인터페이스는 JSON 스키마를 따른다(요청 perception_request / 응답). ROS2 사용 금지. 비ASCII 경로 우회 유지.

#### S3 — 비전 모듈 (M1·M2)
- 목표: 이미지에서 부품의 중심 좌표와 주축 각도를 검출하고(M1), 목표 대비 편차(X/Y/θ)와 보정 벡터를 산출한다(M2). 이 단계에서 **광학 부품(블록/원통 + ArUco 마커)을 씬에 등장**시킨다. 단, 커스텀 모델 작성 금지 제약과 충돌하지 않도록, 부품은 코드에서 mocap/free body로 추가하거나 별도 정당화한다(P1에서 방법 확정).
- 입력: S2 인터페이스, 카메라 이미지.
- 산출물: `/external_module/m1_pose.py`, `/external_module/m2_align.py`.
- 제약: M1은 ArUco 마커 검출을 1차, OpenCV 컨투어를 보조로 사용. 검출 실패 시 명시적 에러 반환.

#### S4 — 보정 폐루프
- 목표: 집기→이송→(측정→편차→보정 이동)→허용오차 판정→배치의 전체 1사이클을 완성한다. 보정 전후 오차를 기록한다.
- 입력: S3 비전 모듈, SRS 허용 오차 임계값(예: ±0.5mm, ±2°).
- 산출물: `/sim/run_cycle.py`, 보정 전/후 오차 로그.
- 제약: 보정 반복 상한(예: 10회) 설정. IK/웨이포인트는 단순하게. 허용 오차 진입 또는 상한 도달 시 종료.

#### S5 — 외란 주입
- 목표: 세 외란을 코드로 주입하는 모듈을 만든다. D1 조명·반사(렌더 조명 변화), D2 카메라 노이즈(가우시안), D3 좌표 드리프트(사이클별 카메라 외부 파라미터 점진 오프셋).
- 입력: S1 렌더 파이프라인, S4 사이클.
- 산출물: `/sim/disturbance.py`(강도 파라미터화), 외란 ON/OFF 토글.
- 제약: 각 외란은 강도(약/중/강)로 조절 가능. D3는 cycle 인덱스에 비례한 오프셋. 우선순위 D1·D2 우선, D3는 여유 시.

#### S6 — 실험·로깅
- 목표: 외란 종류·강도별 + 보정 ON/OFF 조합을 N회씩 반복 실행하고, KPI 4종(잔류 오차, 성공률, 수렴 사이클, 수렴 시간)을 CSV로 누적한다.
- 입력: S4 사이클, S5 외란 모듈, KPI 정의.
- 산출물: `/experiments/run_experiment.py`, `/experiments/results.csv`.
- 제약: 시도 횟수·외란 조건을 CLI 인자로 받는다. 매 시도 `{이미지경로, 정답 pose, 추정 pose, 편차, 외란조건, 사이클, 시간, 성공여부}` 기록.

#### S7 — 리포트 채움
- 목표: S6 결과로 리포트의 자리표시자(`[측정값]` 등)를 실측·그래프·실패모드로 채운다.
- 입력: `/experiments/results.csv`, 리포트 초안(report_draft_v2.md).
- 산출물: 완성된 리포트, KPI 그래프(외란-성능 곡선, 보정 ON/OFF 비교, 정확도-속도 트레이드오프).
- 제약: 5개 필수 항목 누락 없음. SRS 숫자([X]mm 클리어런스, [Y] GSD)를 설정값으로 채움.

## 2. 단계별 평가표 (Rubric)

각 단계는 P3 평가에서 아래 평가표로 채점한다. 항목별 0~2점, 모든 항목 1점 이상 + 총점 합격선(75%) 이상이면 통과.

### 공통 평가 항목 (모든 단계)

| 항목 | 0점 | 1점 | 2점 |
|------|-----|-----|-----|
| 실행 가능성 | 실행 시 에러 | 일부 경고/수동 개입 필요 | 클린하게 실행됨 |
| 산출물 충족 | 산출물 없음 | 일부만 생성 | 정의된 산출물 모두 생성 |
| 제약 준수 | 금지 사항 위반 | 일부 위반 | 모든 제약 준수 |
| 재현성 | 재실행 불가 | 환경 의존 | 동일 입력 → 동일 출력 |

> ★ v2: "실행 가능성"과 "산출물 충족"은 **실제 실행 증거(콘솔 로그·생성 파일)** 가 첨부되어야 2점을 줄 수 있다. 미실행 제출은 두 항목 모두 1점 이하로 채점한다.

### 단계별 추가 평가 항목

**S1 환경 부팅** (★ v2 정정)
| 항목 | 합격 기준 |
|------|-----------|
| 모델 로드 | Franka 공식 모델이 에러 없이 로드 |
| 카메라 렌더 | RGB(+depth) 이미지가 파일로 저장되고, **상단 시점에 로봇이 보임**(로봇 픽셀 비율 정량 확인). 베이스 씬에 광학 부품이 없으므로 로봇 가시성을 기준으로 한다. 부품 배치는 S3에서 다룸 |
| 그리퍼 제어 | 개폐 시 손가락 간격 변화가 **수치로 확인되고(방향 포함)**, 열림/닫힘 RGB가 저장됨 |

**S2 연동 배관**
| 항목 | 합격 기준 |
|------|-----------|
| JSON 왕복 | 요청·응답 스키마가 정의와 일치 |
| 데이터 무결성 | 전송 이미지와 수신 좌표가 손실 없이 연결 |
| 분리 구조 | 외부 모듈과 시뮬이 별도 파일/프로세스로 분리 |

**S3 비전 모듈**
| 항목 | 합격 기준 |
|------|-----------|
| 검출 정확성 | 알려진 부품 위치를 허용 오차 내로 검출 |
| 편차 산출 | X/Y/θ 편차가 올바른 부호·크기로 계산 |
| 실패 처리 | 검출 실패 시 크래시 없이 에러 반환 |

**S4 보정 폐루프**
| 항목 | 합격 기준 |
|------|-----------|
| 루프 완결 | 집기~배치 전 과정이 끊김 없이 1회 완주 |
| 오차 감소 | 보정 후 잔류 오차가 보정 전보다 감소(수치 확인) |
| 종료 조건 | 허용 오차 진입 또는 상한 도달 시 정상 종료 |

**S5 외란 주입**
| 항목 | 합격 기준 |

| 강도 조절 | 약/중/강이 단조적으로 효과 증가 |
| 토글 무결성 | 외란 OFF 시 baseline과 동일하게 복귀 |

**S6 실험·로깅**
| 항목 | 합격 기준 |
|------|-----------|

| 로그 무결성 | CSV가 파싱 가능하고 결측 없음 |

**S7 리포트 채움**
| 항목 | 합격 기준 |
|------|-----------|
| 자리표시자 제거 | `[측정값]` 등이 실측으로 모두 교체 |
| 그래프 정합 | 그래프가 CSV 데이터와 일치 |

## 3. 프롬프트 설정 (독립 작동)

아래 4개 프롬프트는 각각 독립적으로 작동한다. 사용 시 `{...}` 부분을 현재 단계 정보로 치환하여 통째로 입력한다.

> 사용 흐름: 한 단계당 P1 → P2 → P3 → P4 순. P3 통과면 P4에서 다음 단계 P1로. 미달이면 P4 피드백을 P2에 다시 넣어 재작업.

### 3.1 — P1 · 단계 계획 수립 프롬프트

```
역할: 너는 로봇 시뮬레이션 자동화 프로젝트의 기술 기획자다.

배경(자체 포함):
- 프로젝트는 MuJoCo에서 Franka Panda 로봇팔이 광학 부품을 정밀 정렬하는 작업을,
  외부 비전 보정 모듈과 연동하여 구현하고, 현실 외란(조명·노이즈·좌표 드리프트)에 대한
  보정 루프의 강건성을 측정하는 것이다.
- 외부 모듈 ↔ 시뮬은 JSON으로 통신한다(이미지 요청 → 좌표/보정 응답).
- 저장소 구조: /external_module, /sim, /experiments, /report, /docs.
- 이 PC는 비ASCII(한글) 경로 환경이다: 모델 로드는 chdir+ASCII 파일명, 이미지 저장은
  cv2.imencode+파이썬 파일IO 로 우회해야 한다.

현재 단계 정의:
{S_n의 목표 / 입력 / 산출물 / 제약 — 위 1장에서 해당 단계 블록을 그대로 붙여넣기}

직전 단계 산출물 요약(있으면):
{직전 단계에서 만들어진 파일명·핵심 함수 시그니처. 부록 B의 확정 사실도 함께. 없으면 "없음(시작 단계)"}

요청:
1. 이 단계를 구현하기 위한 작업을 3~6개의 구체적 하위 작업으로 분해하라.
2. 각 하위 작업의 입력·출력·핵심 함수 시그니처(이름, 인자, 반환)를 명시하라.
3. 만들 파일 목록과 각 파일의 책임을 한 줄로 적어라.
4. 이 단계에서 발생할 수 있는 기술적 리스크 1~2개와 회피책을 적어라.
5. 코드를 작성하지 마라. 계획만 출력하라.

출력 형식:
## 하위 작업
## 함수 시그니처
## 파일 목록
## 리스크와 회피책
```

### 3.2 — P2 · 코드 제작 프롬프트 (★ v2: 실제 실행 의무화)

```
역할: 너는 로봇 시뮬레이션 자동화 프로젝트의 구현 엔지니어다.

배경(자체 포함):
- MuJoCo + Franka Panda. 외부 비전 모듈은 별도 Python으로 분리.
- 외부 모듈 ↔ 시뮬 JSON 통신. 라이브러리는 mujoco, numpy, opencv-python 중심, ROS2 미사용.
- 저장소 구조: /external_module, /sim, /experiments, /report, /docs.
- 이 PC에서 직접 실행이 가능하다. 코드를 쓰는 데서 멈추지 말고 실제로 돌려라.
- 비ASCII(한글) 경로 환경: 모델 로드는 chdir+ASCII 파일명, 이미지 저장은
  cv2.imencode+파이썬 파일IO 로 우회한다(이미 S1 boot.py에 구현됨, 그 패턴을 재사용).

구현할 계획(P1 산출물):
{P1이 출력한 하위 작업·함수 시그니처·파일 목록을 그대로 붙여넣기}

직전 코드(있으면):
{이전에 만든 관련 파일 내용 또는 "R0B/sim/boot.py 등 기존 파일을 읽어서 재사용". 없으면 "없음"}

피드백(재작업이면):
{P4가 출력한 개선 지시. 최초 작성이면 "없음(최초 작성)"}

요청:
1. 계획에 따라 실행 가능한 코드를 작성하라.
2. 각 파일은 단독 실행 또는 import가 가능해야 한다. 진입점에 간단한 self-test를 포함하라.
3. 외부 의존성은 위 라이브러리로 한정하라. 제약을 위반하지 마라.
4. 코드 상단에 파일 책임과 실행 방법을 주석으로 적어라.
5. 파일별로 전체 코드를 출력하라(부분 수정이 아니라 완성된 파일 단위).
6. [필수] 코드를 작성한 뒤 이 PC에서 실제로 실행하라. 콘솔 로그(검증 출력·PASS 메시지)와
   생성된 산출물(파일 목록·존재 확인)을 결과에 첨부하라. 실행 불가 환경이면 그 사실과
   차단 원인(어느 단계 어떤 에러)을 명시하라. 미실행 상태로 끝내지 마라.

출력 형식:
### 파일: {경로}
(전체 코드)
### 실행 방법
### 실행 결과 (콘솔 로그 + 생성된 산출물 목록)
```

### 3.3 — P3 · 코드 평가 프롬프트

```
역할: 너는 로봇 시뮬레이션 자동화 프로젝트의 코드 리뷰어다. 너는 코드를 작성한 사람과
다른 관점에서, 엄격하게 평가한다.

현재 단계 정의:
{S_n의 목표 / 산출물 / 제약 — 위 1장에서 해당 단계 블록을 그대로 붙여넣기}

평가표(이 단계의 루브릭):
{위 2장에서 공통 평가 항목 + 해당 단계 추가 평가 항목을 그대로 붙여넣기}

평가 대상(P2 산출물 — 코드 + 실행 결과):
{P2가 출력한 전체 코드와 실행 결과(콘솔 로그·산출물 목록)를 그대로 붙여넣기}

요청:
1. 평가표의 각 항목을 0/1/2점으로 채점하고, 각 점수의 근거를 한 줄로 적어라.
2. "실행 가능성"·"산출물 충족"은 실제 실행 증거가 첨부됐을 때만 2점을 줄 수 있다.
   실행 증거가 없으면 두 항목을 1점 이하로 채점하라.
3. 합격선은 만점의 75%이며, 모든 항목이 1점 이상이어야 통과다. 합격/불합격을 명시하라.
4. 불합격이면, 가장 점수가 낮은 항목 순으로 구체적 문제점을 적어라(파일·함수·라인 수준).
5. 코드를 직접 고치지 마라. 점수를 후하게 주지 마라. 의심스러우면 낮게 채점하라.
6. 단, 단계 정의·평가표에 없는 요구(예: 해당 단계에 존재하지 않는 부품)를 사유로 감점하지 마라.

출력 형식:
## 채점표
| 항목 | 점수 | 근거 |
## 총점 및 판정
## 주요 문제점 (불합격 시, 우선순위순)
```

### 3.4 — P4 · 피드백 및 개선/종료 프롬프트

```
역할: 너는 로봇 시뮬레이션 자동화 프로젝트의 기술 리드다. 평가 결과를 받아
다음 행동을 결정한다.

현재 단계: {S_n 이름}
반복 횟수: {이 단계에서 P2~P3를 돈 횟수}, 최대 허용 3회

평가 결과(P3 산출물):
{P3가 출력한 채점표·판정·문제점을 그대로 붙여넣기}

요청:
- 판정이 "통과"이면:
  1. 이 단계의 산출물을 한 줄로 요약하라(다음 단계 P1의 "직전 산출물 요약"에 쓰일 형태).
     이때 다음 단계가 재사용할 핵심 사실(파일명·함수 시그니처·확정된 모델 상수 등)을 포함하라.
  2. 다음 단계로 진행하라고 선언하라. 추가 코드 지시는 하지 마라.

- 판정이 "불합격"이고 반복 횟수 < 3이면:
  1. P3가 지목한 문제점을, P2가 바로 실행할 수 있는 구체적 수정 지시로 변환하라
     (무엇을, 어느 파일·함수에서, 어떻게).
  2. 우선순위순으로 번호를 매겨라. 실행·산출물 관련 문제를 최우선에 둔다.
  3. 이 지시는 P2 코드 제작 프롬프트의 "피드백" 항목에 그대로 들어간다.

- 판정이 "불합격"이고 반복 횟수 >= 3이면:
  1. 남은 문제를 "한계"로 정리하라(리포트 한계 섹션에 들어갈 형태).
  2. 이 단계를 부분 완료로 마감하고 다음 단계로 넘기라고 선언하라.

출력 형식:
## 판정
## (통과 시) 산출물 요약 / (재작업 시) 수정 지시 / (한계 시) 한계 기록
## 다음 행동
```

## 4. 전체 진행 절차 요약

```
for 단계 S in [S1 ... S7]:
    P1(S) → 계획
    반복 r = 0
    while True:
        P2(계획, 직전코드, 피드백) → 코드 + 실제 실행 결과
        P3(S정의, 평가표, 코드+실행결과) → 채점·판정
        r += 1
        P4(판정, r) →
            통과:   산출물 요약(+재사용 사실) → break (다음 단계로)
            불합격 & r<3:  수정 지시 → P2 피드백에 투입 (재작업)
            불합격 & r>=3: 한계 기록 → break (부분 완료, 다음 단계로)
```

---

## 부록 A — 빠른 참조

- **P4 분기:** 통과→산출물 요약을 다음 P1에 / 불합격&r<3→수정 지시를 P2에 / 불합격&r≥3→한계 기록
- **실행 의무:** P2는 항상 실제 실행 + 로그·산출물 첨부. 미실행 제출 금지.
- **비ASCII 경로:** 모델 chdir+basename, 저장 imencode+파일IO.
- **오채점 방지:** P3는 단계에 없는 요구(부품 등)로 감점하지 않는다.
- **한 번에 한 카드:** P1→P2→P3→P4를 따로 입력. 한꺼번에 붙이지 말 것.
